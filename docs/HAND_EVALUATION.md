# テキサスホールデムのハンド評価（7 枚 -> 最強 5 枚）

この文書は、Java のドメイン層で実装するハンド評価ロジックを定義する。

## 1. 入力 / 出力
- 入力: 7 枚のカード（hole 2 枚 + community 5 枚）
- 出力:
  - `Rank`: ハンドカテゴリ（high card から straight flush まで）
  - `Key[]`: 強さ比較用のタイブレーク値。優先度の高い順に並べる
  - `Best5`: 勝ち手を構成する 5 枚（表示 / デバッグ用）

Java での推奨モデル:
- `enum HandRank`
- 不変な `HandValue`
- `rank` と辞書順の `key` に基づく comparator

## 2. 役の順序（強い順）
比較時の順序は次の通り。
1. ストレートフラッシュ
2. フォーカード
3. フルハウス
4. フラッシュ
5. ストレート
6. スリーカード
7. ツーペア
8. ワンペア
9. ハイカード

内部表現では、数値が大きいほど強い値にする（例: Straight Flush = 9）。

## 3. カードのランク
- 数値ランクは 2..14 とする（A = 14）
- A-2-3-4-5 の straight では、Ace は 1 としても扱える

## 4. タイブレークキー定義
`Key[]` は左から順に辞書順で比較する。

### ストレートフラッシュ
- `Key = [HighStraightRank]`

### フォーカード
- `Key = [QuadRank, Kicker]`

### フルハウス
- `Key = [TripsRank, PairRank]`

### フラッシュ
- `Key = [r1, r2, r3, r4, r5]`
  flush suit の上位 5 ランク

### ストレート
- `Key = [HighStraightRank]`

### スリーカード
- `Key = [TripsRank, K1, K2]`

### ツーペア
- `Key = [HighPair, LowPair, Kicker]`

### ワンペア
- `Key = [PairRank, K1, K2, K3]`

### ハイカード
- `Key = [r1, r2, r3, r4, r5]`

## 5. 評価順序
次の順で判定し、最初に成立した役を返す。
1. ストレートフラッシュ
2. フォーカード
3. フルハウス
4. フラッシュ
5. ストレート
6. スリーカード
7. ツーペア
8. ワンペア
9. ハイカード

## 6. 判定戦略
最初に集計を作り、それを各判定で使い回す。
- `rankCounts[15]`: ランクごとの枚数（2..14）
- `suitCounts[4]`: スートごとの枚数
- `cardsByRank[15]`
- `cardsBySuit[4]`

### ストレート判定
- ランクの重複を除いた昇順または降順のリストを作る
- Ace（14）があれば、1 も存在するものとして扱う
- 5 連続のランクを走査し、最も高い straight を返す
- A-2-3-4-5 の場合、high rank は 5 とする

### ストレートフラッシュ
- flush になるスートがある場合、そのスートに限定して straight 判定を行う

### フォーカード
- 枚数が 4 のランクを探す
- kicker は残りの最高ランク

### フルハウス
- 枚数が 3 の中で最大のランクを trips とする
- 残りから、枚数が 2 以上の最大ランクを pair とする

### フラッシュ
- flush suit の上位 5 ランクを採用する

### スリーカード
- 最大の trips ランクを採用する
- 残りから上位 2 枚を kicker とする

### ツーペア
- 上位 2 つの pair ランクを採用する
- 残りの最高ランクを kicker とする

### ワンペア
- 最大の pair ランクを採用する
- 残りの上位 3 枚を kicker とする

### ハイカード
- 全体の上位 5 ランクを採用する

## 7. 比較ルール
比較は次の順で行う。
1. `Rank` を比較する（大きい方が勝ち）
2. `Key[]` を左から順に比較する
3. すべて同じなら引き分け

引き分け時の pot 分配は evaluator の責務ではなく、game / payout ロジック側で扱う。

## 8. 境界条件
- 役も key も同じなら、スートに関係なく引き分け
- A-2-3-4-5 straight は最弱の straight（high = 5）
- evaluator は純粋なドメインロジックとして実装し、DB・HTTP・フレームワークのクラスへ依存させない
